IMDb Worst Movies

Understanding how different variables are related is the foundational goal of any data analysis and it remains true for this project. By firstly understanding the shape of data, and some of the relationships between different variables, we can then begin to model these different aspects and derive an expression that describes what factors most influenced a movies’ placement in IMDb’s 100 Worst Movies list. Furthermore, a secondary goal of this project was to demonstrate not only familiarity with various fundamental statistical techniques, but fluency with them while also applying some more complex methods to create a well-rounded and thoroughly examined piece of data analysis. Therefore, through illustrative visualizations, thorough pre-processing, and model diagnostics, I produced a mathematical expression that describes bad movie rating based on a number of key indicators.

Time-Series Price Analysis

Time-series data is extremely commonplace across any application of data analytics and is particularly powerful for forecasting future values. Nevertheless, there are a multitude of approaches to analyzing this data, each offering different strengths and weaknesses, hence this project aims to compare and statistically evaluate three different models in ascending order of technical complexity. For this use, historical data of diamond and gold prices together with econometric indicators (interest, inflation, federal funds rate) provides a strong foundation for developing a widely applicable model to many other asset prices. Moreover, the use of real world data offers a greater level of complexity and an opportunity to demonstrate a detailed understanding of the analytical tools and techniques necessary to effectively manipulate and model this data. Therefore, by exploring intervariable relationships, transforming and pre-processing the data, developing and forecasting three different models, and continually evaluating performance, I was able to accurately describe and forecast diamond and gold prices with a robust statistical model.

Machine Learning Stock Price Forecasting

For the full code, please visit: https://github.com/jrgdlc/Python-GBM-Stock-Price-Forecast/blob/main/StGBM.py

Click here to try the application

This financial analysis project revolves around the use of a Gradient Boosting Regression model, a form of machine learning. Before going into further detail below, this program works by firstly fetching realtime data then transforming it so that the model is trained to predict future values instead of current values and outputting this forecast in both a table and graph. It is also important to note that any predictions made by the program should not be taken as financial or investment advice.

Firstly, let me explain the model being used. Gradient Boosting Regression models are very powerful yet not all too complicated; they work by creating a large number of simple decision trees which make predictions based on the features, or indicators, they are fed. Individually, these decision trees are not very powerful thus the model will go back and evaluate all of them to see which produces the smallest error relative to the training data. It does this through another decision tree, focusing instead on the errors. With each subsequent decision tree produced, the model will try to minimize the size of the error (residuals) and eventually output a model that best fits the training data. For further information, I found this explanation to be very useful: https://www.geeksforgeeks.org/ml-gradient-boosting/

Gradient Boosting Regression models, as you may infer from the name, are typically used for regressions and thus have limited applicability to time-series data but there are ways around this. Usually with regression models, they take a set of indicators and figure out the relationship to the output and then apply it to a set of indicators they have not yet seen:

In the above image, the model would be trained on columns 1 through 4 and then predict y for column 5. However, what happens when we do not have the X's for column 5? Well the model cannot predict something without the necessary inputs. Therefore, if we are trying to predict the future, the typical regression approach will not work. So in order to account for this, we will have to add autoregressive effects into the model meaning that we have to consider time.

The solution is to, as shown above, shift our y variable to the left. In this case, the model will be trained to use the current indicators to predict the future output. So when we give it today's inputs, it gives us a prediction for tomorrow's output. This is the heart of the problem and was quite challenging to figure out but is a very useful solution when we have a wealth of data and are predicting only a short timespan; generally, the further removed the inputs are from the output, the less reliable our model will be.

This project also prioritized user interaction and usability thus all the major inputs into the model are chosen by the user. This is possible due to the Polygon.ai stock API which fetches realtime data for any stock but due to this, I am unable to host the application live as it requires the input of an API key which should not be shared publicly. The only area which the user does not directly affect is the adjustment of hyperparameters. This is an essential part of any machine learning process and it essentially defines how well the model will fit the data so this is all adjusted automatically within the program however, it is also it's biggest bottleneck and slows down the runtime significantly (as is evidenced in the video above).

The frontend of this application was developed using the streamlit package in Python, which made development and usability much easier. Again, due to the problem with the API key, I could not deploy it to streamlit to have it hosted there. The repository for this project is public on my GitHub profile so please take time to look through it, use it yourself, and potentially add-on to it! As I stated previously, this method is not optimal however, it is a very interesting use-case for this type of model - if you find anything that can be improved please let me know.
